# Object Storage — Instructions

## Storage Abstraction

Blob/object storage is not implemented yet. `GetDownloadUrlHandler` currently returns a placeholder URL, and Engine PDF/blob upload steps are TODOs.

When storage is added, all blob/object storage interactions should go through an adapter interface:

| Environment            | Intended implementation                            |
| ---------------------- | -------------------------------------------------- |
| **Local development**  | Azurite (not in Docker Compose yet)                |
| **Production (Azure)** | Azure Blob Storage                                 |

The active implementation should be selected at startup based on configuration:

```
Storage__Provider = "Azurite" | "AzureBlobStorage"
Storage__ConnectionString = "<connection string>"
Storage__ContainerName = "skillexa"
```

### Adapter Interface Shape

```csharp
public interface IObjectStorage
{
    Task UploadAsync(string key, Stream content, string contentType, CancellationToken cancellationToken = default);
    Task<Stream> DownloadAsync(string key, CancellationToken cancellationToken = default);
    Task<string> GenerateSignedUrlAsync(string key, TimeSpan ttl, CancellationToken cancellationToken = default);
    Task DeleteAsync(string key, CancellationToken cancellationToken = default);
}
```

## Blob Key Layout

| Pattern                  | Content                    | Written By      |
| ------------------------ | -------------------------- | --------------- |
| `pdf/{documentId}.pdf`        | Generated PDF document     | Skillexa-Engine |
| `snapshots/{documentId}.json` | Input data snapshot (JSON) | Skillexa-Engine |

- Keys are flat within the container (pseudo-folders via prefix).
- `documentId` is a BIGINT (auto-increment) — keys are unique within the application.

## Signed Download URLs

- Should be generated by **Skillexa-Core** when the user requests a download (`POST /documents/{id}/download-url`).
- Properties:
  - **Read-only** — the URL permits only `GET` (download).
  - **Single object** — scoped to one specific blob key.
  - **Short TTL** — 5 to 15 minutes.
- Intended implementation: Azure Blob SAS token (both Azurite and Azure Blob Storage support SAS generation).

## Container / Bucket Rules

- The container (e.g., `skillexa`) is **private** — no anonymous/public access.
- Once storage is implemented, all user access to blobs must go through signed URLs issued by the API.
- Never expose storage connection strings or account keys to the client/browser.

## Azurite (Local Development)

- Azurite is not in Docker Compose yet. Add it when blob storage features are implemented.
- Connection string: `DefaultEndpointsProtocol=http;AccountName=devstoreaccount1;AccountKey=<Azurite default key>;BlobEndpoint=http://azurite:10000/devstoreaccount1;`
- The container is auto-created on first upload (or at startup via an init script).
- Azurite supports the same Azure Blob SDK (`Azure.Storage.Blobs`) as production — the same adapter code works in both environments.

## Azure Blob Storage (Production)

- Connection string or Managed Identity–based auth via `Azure.Storage.Blobs`.
- Use a dedicated storage account with a private container.
- Enable soft-delete and versioning as needed for compliance.
